# Web UI

The Meeseeks web app is a Vite + React + TypeScript single-page application living in `src/web/`. It consumes the [server's](server.md) REST API and subscribes to its WebSocket hub for live updates. The build emits `dist/web/`, which the same Fastify process serves statically in production with SPA fallback — there is no separate web server. This React foundation is also what makes a LangChain harness less disruptive than it first appears: LangChain's frontend SDK renders agent state through a React `useStream` hook, so a Meeseeks LangChain adapter would host its renderer inside this existing SPA rather than the xterm.js [console](console.md) — the mapping is analysed in [LangChain frontend rendering](../concepts/langchain-frontend-rendering.md).

## Run modes

In development, `npm run dev` runs Vite on `:5173` and Fastify on `:5174` concurrently; Vite proxies `/api` and `/ws` to Fastify so the SPA can use relative URLs in both modes. In production, `npm run build` produces both `dist/server` and `dist/web`, then `npm start` serves the SPA from Fastify itself. The static handler is registered only when `dist/web` exists, so dev mode (where it doesn't) skips it cleanly.

## Module layout

The `src/web/` tree is organized by responsibility rather than by route:

- `lib/api.ts` — typed `fetch` wrapper. One method per endpoint, sharing request/response types from `src/shared/api.ts`. All thrown errors carry the server's `code` and `message` so callers can show toasts directly.
- `lib/ws.ts` — `WsClient` class with auto-reconnect (exponential backoff to 8s) and a fan-out subscription model. A module-level singleton in `hooks/use-ws.ts` ensures one connection per browser tab.
- `hooks/queries.ts` — React Query hooks. Each query keys on its scope (`['board', boardId]`, `['tickets', boardId, laneName]`, etc.) so the WebSocket invalidator can be precise.
- `hooks/use-ws.ts` — installs a subscription on mount and translates each `WsEvent` into targeted `queryClient.invalidateQueries` calls. Board, lane, and ticket change events invalidate just their scope.
- `hooks/use-runtime-ws.ts` — the runtime-event side of the socket, separate from the query invalidator. It handles `runtime-spawned`/`runtime-status`/`runtime-stdio`/`runtime-message`, mirrors status into `store/runtimes.ts`, and — critically — fans raw `runtime-stdio` bytes (base64-decoded) out to a `Set<StdioHandler>` that terminal panels register against via `onRuntimeStdio`. It also exposes `sendRuntimeInput`/`sendRuntimeResize`, which push keystrokes and PTY dimensions back to the supervisor. This module is the client end of the [Runtime Supervisor](runtime.md)'s byte transport and the single most harness-coupled file in the SPA.
- `routes/` — one file per route. Root (`/`) redirects to `/boards`; boards list (`/boards`), board Kanban (`/boards/:boardId`), board editor (`/boards/:boardId/edit`), ticket detail (`/boards/:boardId/lanes/:laneName/tickets/:filename`). The Board Editor route hosts a left-rail navigation that switches the right pane between One-Shot Prompts, CONTEXT.md, `.claude/skills`, `.claude/bin`, and per-lane configuration.
- `components/console/` — the floating-window subsystem: `Dock` (the taskbar of minimized runtimes), `Mdi` (the multi-document surface that positions panels), `Panel` (a single draggable/resizable window with a status dot, minimize, terminate, and close controls), `xterm-host` (the xterm.js binding — see the byte pipeline below), and `PromptRunModal`. Panel's **close** button removes the window without terminating the runtime — the [dismiss-without-kill](console.md) gesture — while the red **×** calls the terminate mutation.
- `components/` (top level) — `AppShell` (top bar with project name and the [Dock](console.md)), `Sidebar`, `ResizableSplit`, `RuntimeStatusDot`, `MarkdownEditor` and `Markdown` (real markdown editor/renderer — replaced the earlier plain textarea), `SkillsEditor` and `BinEditor` (file-list + editor over the [generic file API](server.md)), `PromptsEditor` (markdown editor + run-log tab for [one-shot prompts](../concepts/one-shot-prompts.md)), modals (`NewBoardModal`, `NewLaneModal`), `Kanban`, `TicketCard`, `Modal`, `ErrorBoundary`, `Logo`.
- `store/ui.ts` — a tiny Zustand store for transient UI state (currently just the selected lane on the board route). `store/mdi.ts` holds console window geometry (per-runtime `x`/`y`/`w`/`h`/`z` and `minimized`), `store/runtimes.ts` mirrors the supervisor's runtime registry, and `store/prompts.ts` accumulates streaming output for one-shot prompt runs keyed by `runtimeId`.

## The console byte pipeline

The terminal is the SPA's one place that renders opaque bytes rather than typed state, and its data path is worth pinning down because it is exactly the seam a [LangChain harness](../syntheses/langchain-as-meeseeks-harness.md) would rewrite. `XtermHost` (`components/console/xterm-host.tsx`) constructs an xterm.js `Terminal` with a `FitAddon`, then wires three flows: (1) on mount it fetches the [ring-buffer](runtime.md) scrollback via `api.getRuntimeSnapshot(runtimeId)` and writes those bytes into the terminal, so a freshly opened panel shows history; (2) it registers an `onRuntimeStdio` handler that calls `term.write(bytes)` for every live `runtime-stdio` chunk addressed to its runtime; (3) it forwards `term.onData` keystrokes through `sendRuntimeInput` and `ResizeObserver`-driven geometry through `sendRuntimeResize` (debounced 100 ms). Nothing in this path inspects the bytes — xterm.js interprets the ANSI itself. That is the whole value and the whole limitation: Meeseeks renders Claude Code's TUI for free but sees none of the structure inside it, which is why lifecycle state is reconstructed out-of-band by the supervisor's hooks rather than parsed here.

## State boundaries

Server state lives entirely in React Query, keyed by scope. Mutations invalidate their own scope on success; the WebSocket invalidator covers everything else, including changes made by other clients or by edits to the filesystem outside the app. Toasts surface errors via `sonner`. UI state — currently just the selected lane — lives in Zustand and does not sync across tabs.

The Kanban view groups tickets into columns by `state.dir` in the order given by `lane.yaml` (which the server returns via `LaneDetail.states`). An "Orphaned" column appears only when at least one ticket has `orphaned === true`, surfacing tickets whose state directory was renamed or removed.

## Refactor surface: what a LangChain frontend would touch

The [frontend-rendering analysis](../concepts/langchain-frontend-rendering.md) argues the rendering gap is narrow because LangChain's `useStream` SDK is TypeScript and lands inside this SPA. Read against the actual module tree, that claim resolves into a specific, small blast radius — most of the SPA is harness-agnostic and would not move.

**What stays untouched.** Everything that renders *project* state rather than *agent* state is indifferent to the harness: the `lib/api.ts` + `lib/ws.ts` transport, the React Query layer (`hooks/queries.ts`, `hooks/use-ws.ts`), all of `routes/`, the `Kanban`/`TicketCard`/`Sidebar` board views, the [focus-gated](../concepts/focus-gated-editor.md) `MarkdownEditor` cluster, and the `Modal`/`ErrorBoundary`/`AppShell` shell. These speak to Meeseeks' own REST/WS surface, which a harness swap does not disturb.

**What is the seam.** The refactor concentrates almost entirely in `hooks/use-runtime-ws.ts` and `components/console/xterm-host.tsx`. Today those two files carry the entire terminal-byte contract: base64 `runtime-stdio` chunks in, xterm `.write()` out, keystrokes and resizes back. A LangChain adapter replaces that contract with a typed stream — `useStream<AgentState>` connected to the [Agent Server](langgraph-agent-server.md) — so `XtermHost` becomes a state renderer (tool-call cards, interrupts, reasoning blocks) instead of a byte sink, and `use-runtime-ws`'s raw-byte fan-out (`onRuntimeStdio`/`sendRuntimeInput`/`sendRuntimeResize`) is retired in favor of the SDK's stream handle. The [component ecosystem](../concepts/langchain-frontend-rendering.md) (AI Elements, assistant-ui, the forkable Agent Chat UI) is what would fill the space xterm.js vacates.

**What survives structurally but changes backing.** The `console/` window system — `Mdi`, `Panel`, `Dock`, `store/mdi.ts` — is genuinely reusable, because its job is *window management*, not rendering: it positions floating panels, tracks z-order and minimized state, and owns the dismiss-vs-terminate distinction. That distinction is the one place the two worlds line up exactly. `Panel`'s **close** (detach the window, leave the runtime running) and **×** (terminate) map one-to-one onto the SDK's `stream.disconnect()` versus `stream.stop()`, which the [frontend-rendering page](../concepts/langchain-frontend-rendering.md) identifies as *join/rejoin is dismiss-without-kill*. The panel chrome would keep working; only the body swaps from `XtermHost` to a `useStream` renderer, and the underlying "keep running while detached" guarantee moves from a live PTY child to a checkpointed server-side thread — strictly more durable than today's [ring buffer](runtime.md), which is lost on restart.

**The one net-new hazard.** `getRuntimeSnapshot` exists only because scrollback is volatile; durable threads make it redundant, so the snapshot-on-mount step in `XtermHost` disappears rather than porting. Conversely, headless tools (agent tools that execute in the operator's browser) are a capability the current architecture has no slot for at all — they would be additive, not a port. Net, the refactor is bounded: two files carry the risk, one subsystem (`console/`) is reused with a new body, and the rest of the SPA is inert to the change.

## Scope and deferrals

The Kanban now supports drag-and-drop between state columns — `TicketCard` is `draggable` and each column registers `onDragOver`/`onDrop` handlers in `Kanban.tsx`, with the active drop target highlighted. The plain-textarea ticket body has been replaced by `MarkdownEditor`, used by ticket bodies, board CONTEXT.md, lane PROCESS.md, skills files, and one-shot prompts. To stop server-driven refetches from clobbering in-progress typing, the ticket body editor and the `FocusGatedMarkdownEditor` wrapper used for CONTEXT.md and PROCESS.md follow the [focus-gated editor pattern](../concepts/focus-gated-editor.md): local state is authoritative while focused or dirty, external edits are adopted on the next clean snapshot rather than blocked permanently, and `MarkdownEditor` exposes the contenteditable's focus transitions through native `focusin`/`focusout` listeners since React's synthetic focus events don't track Milkdown's Crepe surface reliably. UI tests remain deferred.
